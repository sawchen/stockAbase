## 二、采集经验

### 2.1 数据源选择

| 数据源 | 可用性 | 稳定性 | 数据完整性 | 推荐度 |
|--------|--------|--------|-----------|--------|
| **ClickHouse（自有库）** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | **首选** |
| Baostock | 间歇性 | ⭐⭐⭐ | ⭐⭐⭐⭐ | 辅助验证 |
| AKShare (Sina) | 差 | ⭐⭐ | ⭐⭐⭐ | 不推荐 |
| EastMoney | 差 | ⭐⭐ | 差 | 不推荐 |

**核心经验**：自有 ClickHouse 数据库是最可靠的数据源，采集任务应优先从本地获取。

### 2.2 采集策略

1. **增量采集优先**
   - 基于最大时间戳增量采集，避免全量重复
   - 使用 `max(trade_time)` 确定起始点

2. **并发控制**
   - Baostock 有请求频率限制，高并发会被封禁
   - 建议单线程顺序采集，或控制并发在 3-5 个以内

3. **断点续传**
   - 每次采集前记录进度，失败后从断点恢复
   - 使用 watchdog 监控进程状态

4. **数据校验**
   - OHLCV 约束检查：`high >= low`, `close ∈ [low, high]`
   - 与可靠数据源（如 Baostock）抽样比对

### 2.3 常见问题处理

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| Baostock 查询挂起 | 请求频率过高被限流 | 降低并发、添加延时 |
| 数据重复 | 断点续传逻辑缺陷 | 去重 DISTINCT + 主键 |
| 数据缺失 | 采集批次中断 | 标记缺失时间段后续补全 |
| 成交量为0 | 停牌或数据源问题 | 过滤停牌日期 |

### 2.4 复权处理

- **复权方式**：统一使用**后复权**（adjustflag='2'）
- 前后复权数据不可混用，比对时必须保证一致
- ClickHouse 中存储的数据统一为后复权

## 三、技术架构

```
数据采集层
├── Baostock（主要数据源，已停用）
├── AKShare（备用）
└── 自有数据库 → ClickHouse（192.168.31.219）

数据存储层
└── ClickHouse
    ├── stock_info（基本信息）
    ├── stock_minute（主力表，2.33亿）
    ├── stock_minute_backup（备份）
    └── stock_minute_old（历史归档）

数据质量保障
├── 增量采集 + 断点续传
├── OHLCV 约束校验
└── 定期抽样审计（6轮验证完成）
```

## 五、后续建议

1. **数据更新**：每日定时任务增量更新，利用 cron 或 watchdog
2. **监控告警**：数据量异常时触发告警（停牌率、缺失率）
3. **补全计划**：针对已知的缺失时间段制定补全策略
4. **新数据源**：如需补充历史数据，考虑其他可靠数据源
